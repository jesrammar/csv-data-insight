## Resumen ejecutivo (producto)

EnterpriseIQ **sí encaja como producto para consultora de pymes** en forma de MVP avanzado:
- Multiempresa + permisos por cartera (roles `ADMIN/CONSULTOR/CLIENTE`) y acceso por empresa.
- Ingesta CSV/XLSX con límites por plan y recomendaciones “sube por periodos”.
- Módulos operativos que una consultora usa a diario: **Caja**, **Alertas**, **Universal**, **Presupuesto** y **Entregables**.
- “Mes de trabajo” (periodo) centralizado en ajustes por empresa (backend valida `YYYY-MM`).
- Automatización (jobs) y trazabilidad/auditoría de acciones sensibles.
- Observabilidad (Actuator Prometheus + stack Prometheus/Grafana/Alertmanager) y documentación operativa.

Dónde aún no está “redondo” como producto de uso intensivo:
- Falta **QA visual real** (sobre todo 1366×768 / overflow / tablas y gráficas).
- Falta automatización de QA (smoke e2e) y algo de “polish” de UX en casos borde (errores, estados vacíos, cargas grandes).

## Resumen ejecutivo (seguridad/producción)

Buenas bases ya presentes:
- JWT access + refresh con **rotación** y revocación (refresh tokens hasheados en DB, access tokens con `jti` en blacklist). (`backend/.../security/TokenService.java`)
- Refresh token en **cookie `HttpOnly`** con `Secure` y `SameSite` configurables para prod. (`backend/.../controller/AuthController.java`)
- Cabeceras de seguridad y **CSP estricta** en el Nginx del frontend (sin `unsafe-inline` en `style-src`). (`frontend/nginx.conf`)
- Exposición de servicios “sensibles” bound a `127.0.0.1` en prod compose (backend, postgres, prometheus, etc.). (`docker-compose.prod.yml`)

Riesgos/prioridades típicas para cerrar “modo producción”:
- Añadir un terminador TLS (Caddy/Nginx) y HSTS (solo si todo el dominio va por HTTPS).
- Rate limiting / protección de login (fuerza bruta) a nivel Nginx o backend.
- Revisar/asegurar sanitización en generación de HTML de reportes del “assistant” (evitar inyección).

## Hallazgos por área (priorizados)

### P0 (bloqueantes para producción con tráfico real)

1) **TLS + HSTS**
- Estado: el contenedor `frontend` sirve en `:80`. (`docker-compose.prod.yml`)
- Recomendación: terminar TLS en host (Caddy/Nginx) y activar `Strict-Transport-Security` en el terminador.

2) **Protección anti fuerza bruta en login**
- Estado: no se ve rate limit explícito ni lockout por intentos fallidos.
- Recomendación: mínimo un rate limit por IP/usuario en `/api/auth/login` y `/api/auth/refresh` (Nginx o filtro backend).

3) **HTML generado (assistant/reportes)**
- Estado: hay endpoints que generan HTML para reportes del assistant. (`backend/.../controller/AssistantController.java`)
- Recomendación: asegurar que la construcción de HTML aplica escaping/sanitización consistente; evitar que inputs de usuario terminen como HTML sin limpiar.

### P1 (muy importantes para consultoras: horas/día de uso)

1) **QA visual real (responsive/overflow)**
- Estado: hay guardrails de CSS, pero falta ver en navegador los 2–3 overflows “reales” (tablas/gráficas/dashboards).
- Recomendación: checklist manual + arreglos quirúrgicos por pantalla; documentar en `docs/visual-qa-1366x768.md`.

2) **UX de errores y estados vacíos**
- Estado: API devuelve errores como texto (el frontend muestra mensajes). (`frontend/src/api.ts`)
- Recomendación: estandarizar errores (códigos/shape JSON) y mapear a UI (por ejemplo: 413/408/400 con “qué hacer”).

3) **Tests E2E mínimos**
- Estado: backend tiene tests; frontend no tiene suite e2e. (`backend/src/test/...`, no hay `*.test.*` en frontend).
- Recomendación: Playwright (smoke) para flujos: login → overview → import → report.

### P2 (mejoras de calidad/performance)

1) **Dependencias / audit**
- Estado: `docs/known-issues.md` documenta 2 moderadas (Vite/esbuild) por migración a Vite 6.
- Recomendación: plan de upgrade a Vite 6 cuando el producto salga del modo TFG.

2) **Chunk size**
- Estado: `npm run build` avisa de chunks > 500kB (charts/vendor).
- Recomendación: code splitting/manualChunks por áreas (charts/universal/pdf) si afecta a TTI.
